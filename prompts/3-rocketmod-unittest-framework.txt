* The tests must be ran inside a RocketMod Unturned server instance.

  1. Write a Containerfile for a steamcmd based Unturned server with RocketMod installed. Use steamcmd to install the dedicated Unturned server and copy RocketMod from the Unturned Extras directory to the server directory. It must be embedded into the test assembly as an embedded resource. Add a PowerShell command (like Add-Migration) to build / update the Containerfile.
  
  2.  Create RocketModPluginTestBase class that will be the base class for all tests and will handle the container management, server instance management, plugin management and test orchestration.
  
  3. Use Harmony to hook into the server loading process and to trigger / inject code after the test plugin is loaded.Add all Unturned dedicated server cli flags and options as extension methods to a testcontainers builder and allow tests to overwrite the default container setup. Also allow defining an arbitary amount of additional containers like e.g. MySQL or Redis containers if needed by tests. Provide a built-in way of adding a MySQL container with a clean way of getting the connection string for it. Add a method for running code on cleanup after each test and auto up MySQL if its used. Do this by registering callbacks.
  
  4. Define a custom runner for running unit tests. Create a separate nuget package if required. When tests are triggered, first ensure the Containerfile gets built if it doesn't exist already (as vibeplugins:rocketmod), then create an instance with all related resources mount a unique temporary path as RocketMod directory to it and copy the test plugin and it's libraries into each related directory. Then start the container with the configured arguments (add -SkipAssets added by default but make it overrideable). You must wait for the plugin to get loaded in the test container before starting to orchestrate the tests. Use harmony to patch related functions to interact remotely with the tests. You must also provide a way to run code on the server instance and get the result back to the test method, as well as a way to copy files to the container instance and run commands inside the container instance.
 
  5. Provide mock helpers / APIs that are compatible with the Unity runtime. They should be generic and for example allow creating e.g. Zombies or Players on the fly and to simulate actions with or on them.
  
  6. Provide a helper to wait for the next (or an arbitary amount of) ticks / frames / physics updates inside a test. Also add a way to wait for a specific event with an optional expression as matcher / correct instance checker.

Clear the database as well but allow tests to override this or to define their own additional cleanup behaviour. Reuse everything else like the containers themselves to speed up tests.  Make sure to properly handle the case where the server or any plugin fails to load or crashes during loading.
